Tout ce qu'il faut pour
commencer à auditer

Un outil pensé pour passer de “je ne sais pas par où commencer” à un plan d'action concret.

🧩

Extension Chrome

Audit en contexte directement dans votre navigateur, pour comprendre rapidement les points bloquants d'une page.

⚙️

CLI Playwright

Audit automatisé sur une ou plusieurs pages, intégrable en CI/CD, avec rapports exploitables par les équipes produit et dev.

Transparence du modèle

Code open-source, logique de contrôle lisible, couverture détaillée par critère: vous savez ce que l'outil sait faire et ne sait pas faire.

Fonctionnement

Comment lire un
pré-audit d'accessibilité (RGAA)

Chaque étape sert à objectiver un risque. Le rapport aide à prioriser, pas à certifier. Le temps d'audit varie selon le nombre de pages.

  1. 01

    Chargement de la page

    Avec attente de stabilisation du DOM — compatible avec les SPA React/Vue/Angular.

  2. 02

    Audit DOM automatique

    13 thèmes d'accessibilité scannés (référentiel RGAA) : images, couleurs, structure, formulaires, navigation…

  3. 03

    Navigation clavier (Tab ×15)

    Chaque élément focusé est analysé — outline CSS, visibilité, ordre de tabulation.

  4. 04

    Zoom 200% + sans CSS

    Détection de débordements, vérification de la lisibilité sans feuilles de style.

  5. 05

    Test responsive 320px

    Redimensionnement à 320px de large — standard mobile WCAG.

  6. 06

    Screenshots automatiques

    Captures à chaque étape clé pour documenter l'audit.

  7. 07

    Calcul du taux de conformité

    Indicateur de progression: il aide à comparer, mais ne remplace pas un audit de conformité officiel.

  8. 08

    Rapport actionnable

    Sortie JSON/HTML/CSV + rapport vulgarisé avec priorités, impacts utilisateurs et pistes de correction.

Couverture des règles d'accessibilité

Ce que l'outil vérifie,
et ce qu'il ne peut pas certifier

Pour le grand public, le plus important est de savoir comment lire le résultat: aide au diagnostic, pas verdict légal.

Détail complet par règle : RGAA-mapping.md.

Cette transparence permet de savoir exactement comment interpréter les résultats.

Pour une version pédagogique, consultez aussi le guide des règles d'accessibilité (RGAA).

✅ Vérifié automatiquement

L'outil repère des points techniques fréquents: structure HTML, labels, contrastes, navigation clavier de base.

  • Utile pour gagner du temps rapidement
  • Donne des priorités concrètes de correction

⚠️ À confirmer humainement

Certains signaux sont détectés, mais leur conformité dépend du contexte réel d'usage.

  • Exemple: pertinence d'un texte alternatif
  • Exemple: qualité d'une aide de formulaire

❌ Nécessairement manuel

Une partie des règles d'accessibilité (RGAA) nécessite un jugement humain, des tests d'usage et parfois des technologies d'assistance.

  • Le pré-audit ne remplace pas un audit certifié
  • Il sert de point de départ fiable pour agir

Installation

Commandes simples,
rapports immédiats

Deux usages fréquents côté CLI: auditer une home page, puis élargir à plusieurs pages.

⚙️

CLI (recommandé)

Audit automatisé, CI/CD, rapports + vulgarisation

$git clone https://github.com/Stan69000/rgaa-audit
$cd rgaa-audit/cli
$npm install
$npx playwright install chromium
# 1) Rapport home page
$node bin/rgaa-audit.js https://mon-site.fr -o json -s ./rapport-home.json --vulgarized-save ./rapport-home-vulgarise.html
# 2) Rapport multi-pages
$node bin/rgaa-audit.js https://mon-site.fr --depth 8 -o json -s ./rapport-multipages.json --vulgarized-save ./rapport-multipages-vulgarise.html
